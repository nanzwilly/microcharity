[Heading1] 1. Introduction
[Heading2] 1.1 Purpose
This Software Requirements Specification (SRS) document defines the functional and non-functional requirements for rebuilding the MicroCharity crowdfunding portal (www.microcharity.com). The new system will replace the current WordPress-based implementation with a custom-built, scalable web application tailored specifically to MicroCharity's mission of raising and disbursing funds for educational and medical needs of underprivileged individuals.
This document is intended as the authoritative reference for all development, design, testing, and project management activities related to the portal rebuild.
[Heading2] 1.2 Scope
The MicroCharity Portal Rebuild encompasses the following high-level scope:
[ListParagraph] A public-facing website with dynamic home page, cause listing, individual cause donation pages, and static informational content.
[ListParagraph] An administrator and content manager backend dashboard for managing donation pages, tracking donations, managing donors, and generating reports.
[ListParagraph] Integration with Razorpay for online payment processing.
[ListParagraph] Integration with Gmail for Business for transactional and campaign email delivery.
[ListParagraph] Donor management, donation receipts (PDF), and role-based user access control.
[Heading2] 1.3 Definitions & Acronyms

[Heading2] 1.4 Overview of Current System
The current MicroCharity portal runs on WordPress using a charity-focused theme. While functional, it presents limitations in customization, performance, and maintainability. The new system aims to address these constraints while preserving all existing content and improving the user experience for donors, administrators, and content managers.
NOTE
All existing static pages, images, and content from www.microcharity.com must be migrated to the new system with no content loss.
[Heading1] 2. Overall System Description
[Heading2] 2.1 Product Perspective
The new MicroCharity portal is a purpose-built web application that replaces the existing WordPress installation. It will operate as a standalone system with the following key integrations:
[ListParagraph] Razorpay — Payment gateway for processing online donations
[ListParagraph] Gmail for Business (SMTP / Gmail API) — Transactional emails, donation receipts, and campaign emails
[ListParagraph] Cloud file storage — For featured images, PDF receipts, and document attachments
[Heading2] 2.2 User Classes
User Class
Access Level
Description
Public Visitor
Read-only
Any unauthenticated user browsing the portal, viewing causes, and making donations.
Donor
Limited
Registered donor who can view their donation history via email links or donor portal (optional).
Content Manager
Restricted Admin
Can create, edit, and publish donation pages; view and send receipts; approve offline donations.
Administrator
Full Access
Full system access including user management, settings, reports, templates, and email campaigns.
[Heading2] 2.3 Technology Stack Recommendations
The following technology stack is recommended for the rebuild, though final choices rest with the development team:
[ListParagraph] Frontend: React.js or Next.js (with server-side rendering for SEO)
[ListParagraph] Backend: Node.js with Express or a Python-based framework (Django / FastAPI)
[ListParagraph] Database: PostgreSQL (relational structure suits donor and donation tracking)
[ListParagraph] CMS/Admin Framework: Custom admin dashboard or AdminJS/React Admin
[ListParagraph] Rich Text Editor: TipTap, Quill.js, or CKEditor 5 (WYSIWYG HTML)
[ListParagraph] Payment: Razorpay Node.js / Python SDK
[ListParagraph] Email: Gmail API with OAuth 2.0 or SMTP relay
[ListParagraph] PDF Generation: Puppeteer, WeasyPrint, or PDFKit for donation receipts
[ListParagraph] Hosting: AWS / GCP / DigitalOcean with SSL certificate
[ListParagraph] File Storage: AWS S3 or Cloudinary for images and PDF storage
[Heading2] 2.4 Constraints & Assumptions
[ListParagraph] All monetary values will be in Indian Rupees (INR).
[ListParagraph] The Razorpay payment gateway will be configured for Indian domestic payments.
[ListParagraph] PAN collection is mandatory for donations above INR 10,000 per Indian income tax regulations.
[ListParagraph] Email delivery will use Gmail for Business account associated with MicroCharity.
[ListParagraph] The system must be mobile-responsive and optimized for low-bandwidth connections.
[ListParagraph] All existing URLs from the current WordPress site should be redirected appropriately (301 redirects).
[ListParagraph] SSL/HTTPS is mandatory for all pages.

[Heading2] 8.2 Razorpay Donation Flow
The following describes the end-to-end online donation flow:
[ListParagraph] Visitor opens a Donation Page on the public portal.
[ListParagraph] Visitor selects or enters a donation amount and clicks 'Donate Now'.
[ListParagraph] System creates a Razorpay Order via API (stores order_id with status 'created').
[ListParagraph] Razorpay Checkout modal opens in the browser for the visitor to complete payment.
[ListParagraph] On payment success, Razorpay redirects to success URL and triggers a webhook to the portal backend.
